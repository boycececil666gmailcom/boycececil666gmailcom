# Unity IL2CPP Build Structure — What Every File and Folder Does

When you build a Unity project for Windows using the **IL2CPP scripting backend**,
the output looks very different from a Mono build. Instead of shipping your C# code
as managed `.dll` files that run on a JIT runtime, IL2CPP ahead-of-time compiles
everything into a single native binary. Understanding what each piece of the output
does — and which parts to ship — is essential for debugging, profiling, and
distributing your game correctly.

---

## Root-Level Files

| File | Purpose |
|------|---------|
| `YourGame.exe` | The entry point. A thin launcher that bootstraps the Unity engine and then delegates to `UnityPlayer.dll`. |
| `UnityPlayer.dll` | The Unity engine runtime — rendering, physics, asset loading, the scripting host, and all engine subsystems live here. |
| `GameAssembly.dll` | **The IL2CPP output.** All your C# game code (and every third-party managed library) compiled ahead-of-time into native x64 machine code. This replaces the .NET JIT entirely. |
| `baselib.dll` | Unity's platform abstraction layer — low-level threading, memory allocation, atomics, and OS primitives. |
| `UnityCrashHandler64.exe` | A watchdog process that catches unhandled crashes and writes minidump files for post-mortem analysis. |
| `WinPixEventRuntime.dll` | Microsoft PIX integration — injects GPU timing and event markers so you can profile the game in PIX for Windows. |

---

## `D3D12/`

Contains `D3D12Core.dll` and `d3d12SDKLayers.dll` — Microsoft's redistributable
**DirectX 12 Agility SDK** runtime. Unity ships these directly so it can target a
newer D3D12 feature set than whatever the OS currently has installed, without
requiring Windows updates from players.

---

## `YourGame_Data/` — Runtime Data

Everything the running game needs. This folder ships with the game.

### Scene and Asset Files

| File | Purpose |
|------|---------|
| `boot.config` | Key-value startup settings for the Unity player (graphics API preference, Burst options, etc.). |
| `app.info` | Company name and product name strings. |
| `globalgamemanagers` / `.assets` | Serialized Unity manager objects: Quality Settings, Input Manager, Physics Settings, Tags, Layers, etc. |
| `level0` | The serialized data for your first scene — all GameObjects, components, and asset references. |
| `sharedassets0.assets` | Shared asset bundles referenced across scenes (materials, textures, meshes). |
| `resources.assets` | Assets placed in a `Resources/` folder, loadable at runtime via `Resources.Load()`. |
| `RuntimeInitializeOnLoads.json` | Manifest of methods decorated with `[RuntimeInitializeOnLoad]` that Unity calls at startup. |
| `ScriptingAssemblies.json` | Index of all managed assemblies and their IL2CPP-compiled type mappings. |

### `il2cpp_data/Metadata/global-metadata.dat`

This is one of the most important files in an IL2CPP build. Because IL2CPP compiles
C# to native code, type names, method names, field names, and all other reflection
metadata are stripped from `GameAssembly.dll`. They are stored here instead, in a
binary database. At runtime, `GameAssembly.dll` reads this file to support
`typeof()`, `GetType()`, serialization, and any other reflection-dependent feature.

> **Security note:** `global-metadata.dat` is the primary target for Unity game
> reverse-engineering tools such as Il2CppDumper. For released games, consider
> encrypting or obfuscating this file.

### `Plugins/x86_64/lib_burst_generated.dll`

All code compiled by Unity's **Burst compiler** — typically jobs, math-heavy loops,
and ECS systems. Burst produces highly optimised native SIMD code (AVX2/SSE4) that
is separate from the IL2CPP output. This is a native DLL; it does not contain IL.

## `YourGame_BackUpThisFolder_ButDontShipItWithYourGame/` — Build Intermediates

Unity keeps these for faster **incremental rebuilds**. They are not needed by the
shipped game and should be excluded from distribution packages. The folder name
says it all.

### `il2cppOutput/`

The **C++ source files** generated by IL2CPP from your managed assemblies:

- One `.cpp` file per input assembly (e.g. `Assembly-CSharp.cpp`,
  `UnityEngine.CoreModule.cpp`).
- Corresponding `_CodeGen.c` files contain IL2CPP registration tables.
- `GenericMethods__N.cpp` files hold AOT-generated instantiations of C# generic
  methods — Unity splits these across many files to keep individual translation
  units manageable for the compiler.

`GameAssembly.dll` was compiled and linked from all of these files.

### `Managed/`

The original **.NET managed `.dll` assemblies** — your C# code and all
dependencies as they existed before IL2CPP processed them. These are the
**input** to the IL2CPP transpiler. Each `.dll` has a corresponding `.pdb`
(debug symbol) file. These are *not* loaded at runtime in an IL2CPP build.

### `GameAssembly.pdb`

Debug symbol file for `GameAssembly.dll`. Maps native x64 instruction addresses
back to C# method names and source line numbers. Required for readable crash stack
traces in development builds; not needed in a stripped release.

---

## `YourGame_BurstDebugInformation_DoNotShip/` — Burst Debug Data

Contains LLDB/WinDbg-compatible debug information for Burst-compiled code. Useful
during development for profiling and crash analysis. Not needed at runtime.

---

## Managed vs Native DLLs — The Key Distinction

The `.dll` file extension is just a container format (PE/COFF). What's *inside* is
completely different depending on who produced it:

| | Managed DLL | Native DLL |
|---|---|---|
| **Produced by** | C# compiler (Roslyn) | C/C++ compiler (MSVC, Clang) |
| **Contents** | IL bytecode (CPU-agnostic) | x64 machine instructions |
| **Execution** | Needs a .NET / Mono runtime + JIT | Loaded directly by the OS — no JIT |
| **PE marker** | Has a CLI header | No CLI header |
| **Inspect with** | ILSpy, dnSpy, dotPeek | Ghidra, x64dbg, dumpbin |
| **Examples in this build** | `BackUp/Managed/*.dll` (build inputs) | `GameAssembly.dll`, `UnityPlayer.dll` |

---

## The IL2CPP Compilation Pipeline

```
C# Source  (.cs files)
    │
    ▼  Roslyn compiler
Managed DLLs  ←─ BackUpFolder/Managed/   [NOT shipped]
    │
    ▼  il2cpp.exe transpiler
Generated C++  ←─ BackUpFolder/il2cppOutput/   [NOT shipped]
    │
    ▼  MSVC / Clang compiler + linker
GameAssembly.dll  ─────────────────────────────  [SHIPPED]
    +
global-metadata.dat  ──────────────────────────  [SHIPPED]
```

The `BackUpFolder` preserves the intermediate steps so Unity can skip re-running
everything on the next build if only a few scripts changed.

## What to Ship vs What to Keep

| Item | Ship? |
|------|-------|
| `YourGame.exe`, `UnityPlayer.dll`, `GameAssembly.dll`, `baselib.dll` | ✅ Yes |
| `UnityCrashHandler64.exe`, `WinPixEventRuntime.dll` | ✅ Yes |
| `D3D12/` | ✅ Yes |
| `YourGame_Data/` (entire folder) | ✅ Yes |
| `BackUpThisFolder_ButDontShipItWithYourGame/` | ❌ No |
| `BurstDebugInformation_DoNotShip/` | ❌ No |
