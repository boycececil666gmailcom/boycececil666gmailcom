# How MCP Servers Work: stdio, JSON-RPC, and Process Communication

This article explains how a Model Context Protocol (MCP) server is structured, how it communicates with an AI assistant using JSON-RPC over stdio, and how it launches and controls external processes like the Godot game engine.

## What is an MCP Server?

An MCP server is a local program that exposes a set of **tools** to an AI assistant (such as GitHub Copilot or Claude). The AI calls these tools by name, passing parameters, and receives structured results back. This gives the AI the ability to interact with external systems — file systems, game engines, compilers, APIs — without the AI itself needing to know how those systems work internally.

The `godot-mcp` server, for example, exposes tools like `launch_editor`, `run_project`, `create_scene`, and `get_debug_output` to let an AI assistant drive the Godot game engine.

## Project Structure

A typical MCP server built with the TypeScript MCP SDK looks like this:

```
godot-mcp/
├── src/
│   ├── index.ts                  ← Main server (TypeScript)
│   └── scripts/
│       └── godot_operations.gd  ← Godot headless script (GDScript)
├── build/                        ← Compiled output (generated by tsc)
│   ├── index.js
│   └── scripts/godot_operations.gd
└── .vscode/mcp.json             ← VS Code MCP configuration
```

TypeScript is compiled to plain JavaScript via `tsc`, then `node build/index.js` runs the server.

## How the AI Talks to the MCP Server: JSON-RPC over stdio

VS Code spawns `node build/index.js` as a **child process**. Communication happens through **stdin/stdout** — the AI sends requests as newline-delimited JSON, and the server replies the same way.

```
VS Code (parent)
    │
    │  stdin  ────► JSON-RPC request
    │  stdout ◄──── JSON-RPC response
    │
node build/index.js (child process)
```

The wire format is **JSON-RPC 2.0** — a simple envelope:

```json
// Request (AI → Server)
{ "jsonrpc": "2.0", "id": 1, "method": "tools/call", "params": { "name": "add", "arguments": { "a": 3, "b": 4 } } }

// Response (Server → AI)
{ "jsonrpc": "2.0", "id": 1, "result": 7 }

// Error
{ "jsonrpc": "2.0", "id": 1, "error": { "code": -32601, "message": "Method not found" } }
```

The `id` field matches responses back to requests, which matters when multiple calls are in-flight simultaneously.

### Minimal Python stdio JSON-RPC Server

```python
# server.py
import sys, json

METHODS = {
    'add':   lambda p: p['a'] + p['b'],
    'greet': lambda p: f"Hello, {p['name']}!",
}

for line in sys.stdin:
    req = json.loads(line)
    method = req['method']
    if method in METHODS:
        response = {'jsonrpc': '2.0', 'id': req['id'], 'result': METHODS[method](req['params'])}
    else:
        response = {'jsonrpc': '2.0', 'id': req['id'], 'error': {'code': -32601, 'message': 'Method not found'}}
    sys.stdout.write(json.dumps(response) + '\n')
    sys.stdout.flush()
```

```python
# client.py
import subprocess, json

server = subprocess.Popen(['python', 'server.py'], stdin=subprocess.PIPE, stdout=subprocess.PIPE, text=True)

def call(method, params, call_id=1):
    server.stdin.write(json.dumps({'jsonrpc': '2.0', 'id': call_id, 'method': method, 'params': params}) + '\n')
    server.stdin.flush()
    return json.loads(server.stdout.readline())['result']

print(call('add',   {'a': 3, 'b': 4}))   # 7
print(call('greet', {'name': 'Bao'}))     # Hello, Bao!
server.stdin.close()
server.wait()
```

The key detail: `sys.stdout.flush()` forces Python to send the reply immediately. Python buffers stdout by default, which would cause the client to wait forever.

## stdio vs TCP/IP vs HTTP

stdio is **always local** — it is a kernel pipe between two processes on the same machine. No network, no port, no IP address.

```
stdio pipe          ← raw bytes, same machine only
Unix domain socket  ← like TCP but via filesystem path
TCP socket          ← full TCP/IP stack, works same or remote machine
HTTP                ← TCP + request/response text envelope
HTTPS               ← HTTP + TLS encryption
JSON-RPC / MCP      ← JSON format convention on top of any transport
```

MCP uses stdio for local tools because it is simpler and faster: the parent process owns both ends of the pipe from birth, so there are no timing issues, port conflicts, or connection failures to handle. The MCP spec also supports HTTP + SSE for remote servers accessible over a network.

## How the MCP Server Talks to Godot: Two subprocess Modes

The MCP server itself spawns Godot as a subprocess in two modes:

**Mode A — Long-running process** (for `launch_editor`, `run_project`):

```
spawn(godot.exe, ['--editor', '--path', projectPath])
  └── stdout/stderr buffered into memory
get_debug_output → reads that buffer → returns to AI
```

**Mode B — One-shot headless script** (for `create_scene`, `add_node`, etc.):

```
execFile(godot.exe, [
    '--headless', '--path', projectPath,
    '--script', 'godot_operations.gd',
    'create_scene',
    '{"root_node_type":"Node2D"}'
])
```

`godot_operations.gd` is a GDScript file that runs inside Godot's headless engine, reads the operation name and JSON parameters from `OS.get_cmdline_args()`, performs the requested scene operation, and exits. Its stdout is captured and returned to the AI as the tool result.

This two-language bridge (TypeScript → GDScript) is only possible via subprocesses — you cannot import a `.gd` file into Node.js directly.

## Node.js Runtime vs Java JVM

Node.js uses the **V8 JavaScript engine** (also used in Chrome). V8 JIT-compiles JavaScript directly to native machine code at runtime. There is no intermediate bytecode format — you distribute plain `.js` source text, and V8 re-compiles it on each machine.

The Java JVM compiles source to platform-independent **bytecode** (`.class` / `.jar`), which the JVM then interprets or JIT-compiles. The bytecode is the distributed artifact.

| | Java | Node.js |
|---|---|---|
| Distributed artifact | `.class` / `.jar` bytecode | `.js` source text |
| Runtime | JVM (separate install) | V8 (bundled inside `node.exe`) |
| Execution | Bytecode → JIT to native | JS text → JIT directly to native |

## Node.js Streams and EventEmitter

Node.js I/O is **non-blocking and event-driven**. HTTP request bodies, process stdout, and file streams all arrive in chunks. The `EventEmitter` pattern lets you subscribe to events as they fire:

```js
req.on('data', chunk => body += chunk);  // fires per chunk
req.on('end',  ()    => handle(body));   // fires when complete
```

`.on()` is just a regular method — internally EventEmitter is a Map of event name → list of callbacks:

```js
emitter.on('line', fn)     // subscribe, fires every time
emitter.once('line', fn)   // subscribe, fires once only
emitter.off('line', fn)    // unsubscribe
emitter.emit('line', data) // trigger manually
```

This is the same concept as browser DOM events (`addEventListener`), just different naming.

## Streaming Responses (SSE)

Instead of waiting for a full response, a server can stream chunks to the client as they are produced — this is how ChatGPT and Claude stream tokens to the screen.

```js
// Server — push each token immediately
res.writeHead(200, { 'Content-Type': 'text/event-stream' });
for (const word of words) {
    res.write(`data: ${word}\n\n`);
    await delay(300);
}
res.end();

// Client — render each chunk on arrival
const reader = res.body.getReader();
while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    process.stdout.write(new TextDecoder().decode(value));
}
```

The MCP server does not stream tool results — tool responses are always complete before being returned to the AI.

## Why Use Subprocesses at All?

Keeping everything in one process is simpler and often the right choice. Subprocesses are needed when:

- **Different language** — Node.js cannot run GDScript; only Godot can
- **Crash isolation** — a subprocess crash does not kill the parent process
- **True parallelism** — Python's GIL limits threads; subprocesses bypass it entirely
- **Reusing existing tools** — calling `git`, `ffmpeg`, or `ping` without rewriting them
- **Security sandboxing** — running untrusted code in an isolated process

On Windows, spawning a subprocess calls the `CreateProcess()` Win32 API. Python's `subprocess.Popen` wraps this behind a cross-platform interface so you do not need to write platform-specific code.
